# 旅游规划输出合规审查

## 目标

本 Skill 用于审查旅游规划协调者 Agent 的工作质量。审查依据是项目中定义的完整工作流程、规划原则、反模式规则和输出格式标准。审查结果必须严格对照本规范，不凭主观偏好扣分。

## 审查总纲

审查分五个维度，每个维度独立评分（0-100），最终取加权综合分：

```json
{
  "workflow_compliance": { "weight": 0.20 },
  "thinking_quality": { "weight": 0.15 },
  "content_depth": { "weight": 0.30 },
  "output_quality": { "weight": 0.20 },
  "laziness_detection": { "weight": 0.15 }
}
```

综合分 < 75 为不合格，必须修正后重新提交。综合分 < 65 为严重不合格，必须大幅修改。

> **核心原则**：形式合规不等于内容合格。一个回答可以格式完美、无违规、步骤齐全，但仍然因为内容泛泛、缺乏深度而不合格。内容深度是最重要的审查维度。

## 一、工作流合规审查（workflow_compliance）

协调者必须按以下七步顺序执行，不可跳步、乱序或合并。

### 步骤完整性检查清单

| 步骤 | 必须执行的事项 | 跳步特征 |
|---|---|---|
| 第一步：解析旅行需求 | 从用户请求中提取目的地、出发地、日期/天数、交通偏好、预算、同行人群、兴趣偏好、必去/不去 | 直接输出方案但 normalized_query 缺少关键字段 |
| 第二步：需求准入与追问 | 首次请求必须判断必须/条件必须/可选信息；缺少必须信息时输出 `insufficient_information: true` 并追问 | 用户首次请求未说"别问"却直接给方案；或追问了但没覆盖所有必须缺口 |
| 第三步：攻略素材采集 | 调用 `zhihu_guide_material`，提取地点、体验信号、负面信号 | answer 中出现攻略类内容（避坑、本地推荐）但无知乎采集记录 |
| 第四步：地理事实验证 | 委托 amap-agent 验证候选 POI、距离、路线 | answer 中有具体距离/时间数据但 route_validation 为空或无 amap-agent 调用记录 |
| 第五步：路线设计 | 先聚类再写行程；每天一主区域；2-4 主停留点；顺路优先于热度 | 一天内出现跨区跳跃；主停留点超过 5 个且用户未要求紧凑 |
| 第六步：输出质量审查 | 提交草稿给 review-agent 审查 | 最终输出未经审查 |
| 第七步：最终输出 | 加载 travel-answer-format，使用小标题范式 | 未使用 `### 地点名` 格式；缺少必要字段 |

### 步骤顺序违规

以下任何一条即为严重违规（critical_issue）：

- 先写行程再验证路线（第五步在第四步之前完成）
- 跳过追问直接给方案（用户未明确说"别问"时）
- 跳过攻略采集直接规划（目的地明确但未调用 zhihu_guide_material）
- 跳过地理验证直接输出（answer 中有具体数据但未委托 amap-agent）
- 跳过审查直接输出最终方案

### 步骤执行质量

即使步骤存在，还需检查执行质量：

**第一步执行质量：**
- normalized_query 是否完整提取了用户请求中的所有已知信息
- 是否遗漏了用户明确提到的偏好、约束或特殊要求

**第二步执行质量：**
- 追问是否一次性覆盖了所有必须缺口（目的地、时间、起点、规划目标）
- 是否附带了最多 2 个可选偏好问题
- 用户回复后是否继续追问（违规：追问只做一轮）
- 用户未说"别问"时是否跳过了追问（违规）

**第三步执行质量：**
- topic 格式是否合理（目的地 + 主题）
- 是否从结果中提炼了：反复出现的地点、本地体验信号、负面信号（拥挤/商业化/排队）、适合人群建议
- 是否将知乎内容当作地理事实使用（违规）

**第四步执行质量：**
- 委托任务是否清晰（给出了具体的验证目标）
- 是否验证了关键路段：起点到第一主区域、跨区域移动、当天返回终点
- 是否避免了重复查询同一问题
- route_validation 记录是否与 answer 中使用的数据一致

**第五步执行质量：**
- 是否先按坐标/区县/距离聚类，再写行程
- 每天是否只围绕一个主区域或慢行走廊
- 是否按慢旅游适配度评分而非热度评分（参考评分权重：route_smoothness 0.30, slow_travel_fit 0.25, differentiation 0.20, interest_match 0.15, content_signal 0.10）
- 热门景点是否标注了拥挤风险并提供更安静替代
- 当知乎热度与高德路线可行性冲突时，是否路线可行性优先

## 二、思考质量审查（thinking_quality）

### thinking_result 必须覆盖的要素

| 要素 | 检查内容 | 缺失特征 |
|---|---|---|
| 需求识别 | 是否正确提取了目的地、天数、交通偏好、人群、预算等 | thinking_result 中未提及用户的关键需求字段 |
| 信息缺口判断 | 是否识别了哪些信息不足、需要追问 | 未分析信息充足性就直接进入采集/规划 |
| 工具/成员调用依据 | 是否说明了为什么要调 zhihu_guide_material 或 amap-agent | 调用工具但未说明调用理由 |
| 路线取舍 | 是否说明了为什么选择这些地点、为什么排除其他地点 | 只列了选择结果，未说明排除理由 |

### 偷懒推理特征

以下任何一条即为偷懒推理（laziness_flag）：

- 跳过信息缺口分析，直接假设默认值（用户未说"按默认"时）
- 省略验证步骤，声称"已验证"但 route_validation 中无对应记录
- thinking_result 内容过于笼统，没有针对具体目的地/用户需求的分析
- 照搬模板句式，未结合实际问题填充内容

### thinking_result 反模式

- 暴露逐字隐藏推理链路或内部实现细节（违规，必须只输出面向用户的摘要）
- 输出无关试探、自我对话或犹豫不决的推理过程
- 与 planning_process 内容高度重复（两者应有区分：thinking_result 侧重推理，planning_process 侧重执行）

## 三、内容深度审查（content_depth）

> 这是最重要的审查维度。形式合规不等于内容合格。

### A. 地点介绍深度

每个地点的"简单介绍"必须提供**对用户决策有实际帮助的信息**，不能只是定义式的一句话。

**不合格示例（泛泛定义）：**
- "大理是云南最受欢迎的慢生活城市"
- "喀什老城是中国唯一保存完整的伊斯兰文化古城"
- "漠河北极村是中国最北点"

**合格示例（有决策价值）：**
- "大理古城人民路适合下午闲逛，两侧有独立书店和手作银饰店；洱海西岸才村码头看日落比双廊安静得多"
- "喀什老城吾斯塘博依巷有百年茶馆，当地人下午喝砖茶、下棋，游客少；周日牛羊巴扎最热闹但气味重，建议带口罩"
- "北极村商业化较重，北红村更原始——俄罗斯族木刻楞房屋、没有路灯的夜晚能看到银河"

**检查标准：**
- 介绍是否回答了"去了看什么/做什么/体验什么"，而不只是"这是什么地方"
- 是否提供了具体可执行的信息（哪条巷子好、什么时间段去、和哪里比较）
- 是否有用户视角的判断（值不值得专门去、还是路过看看即可）

### B. 推荐理由深度

每个地点的"为什么推荐"必须**直接关联该用户的特定需求**，不能是放之四海而皆准的通用推荐语。

**不合格示例（通用推荐语）：**
- "自然风光壮美，值得一去"（对任何用户都成立）
- "适合慢节奏散步"（没有说明为什么对这个用户特别合适）
- "是中国最美公路之一"（名气≠适合这个用户）
- "风景原始震撼"（空洞形容词）

**合格示例（关联用户需求）：**
- "你说喜欢自然多一点，阿尔山的火山天池和温泉是草原行程中难得的地质景观变化，不至于连续看一个月草原审美疲劳"
- "你一个人开越野车，这段戈壁公路虽然荒但路面平整，适合你练长途独驾心态，为后面新藏线无人区做心理准备"
- "10万预算紧张，这个免费的湿地公园比收费景区性价比高，而且就在你当天路线旁边不绕路"

**检查标准：**
- 推荐理由是否引用了用户的特定偏好（自然/预算/一人行/越野车/时间窗口等）
- 是否解释了"为什么对这个用户特别合适"而不只是"这个地方很好"
- 是否结合了路线逻辑（顺路、不绕路、时间窗口合适）

### C. 时间安排深度

时间安排必须具体到**每天做什么、每个阶段如何分配**，不能只给一个总天数范围。

**不合格示例（模糊时间）：**
- "建议分3-5天"（为什么是3天还是5天？这2天差别在哪？）
- "建议停留7-10天"（10天里每天干什么？）
- "建议分1天走完"（9小时纯驾驶，怎么休息？在哪过夜？）

**合格示例（具体时间）：**
- "全程9小时纯驾驶，建议分2天：第一天开到锦州（4.5h），下午逛笔架山；第二天锦州到丹东（4.5h），下午到鸭绿江断桥看日落"
- "喀什建议4天：第1天老城+艾提尕尔清真寺+茶馆；第2天大巴扎+牛羊巴扎（周日）；第3天香妃园+高台民居；第4天休整补给"
- "漠河周边5天：第1天北极村；第2天北红村（更原始，推荐住一晚）；第3天龙江第一湾+乌苏里浅滩；第4天漠河县城补给；第5天出发去根河"

**检查标准：**
- 多天行程是否逐天分配了主题或重点
- 长距离移动是否说明了分几天、每天开多远、在哪过夜
- 是否考虑了驾驶疲劳、补给需求、景点开放时间等实际因素

### D. 路线细节深度

路线描述必须提供**可执行的导航信息**，不能只给起终点。

**不合格示例（只有起终点）：**
- "距北京约841km。推荐G1京哈高速→G11鹤大高速，全程高速"
- "距喀什约300km。中巴友谊公路（G314），帕米尔高原"
- "距抚远约700km。沿黑龙江（界河）西行"

**合格示例（可执行细节）：**
- "北京→丹东841km/9h：G1京哈高速→秦皇岛服务区休息（4h处，有肯德基和加油站）→G11鹤大高速→丹东出口。过路费约359元。建议早6点出发避开北京早高峰，下午3点到丹东"
- "喀什→塔县300km/5h：G314中巴友谊公路，白沙湖在155km处（可停车拍照），喀拉库勒湖在190km处（看慕士塔格峰倒影的最佳角度在湖西侧），塔县检查站在280km处需下车登记身份证"
- "叶城→三十里营房360km/7h：出叶城后连续爬升，库地达坂（3150m）在60km处，麻扎达坂（4969m）在180km处——这是新藏线第一个5000m级挑战，建议在麻扎达坡顶停车适应10分钟再继续"

**检查标准：**
- 是否标注了关键途经点、服务区、加油站在哪
- 长距离路段是否标注了分段距离和预计耗时
- 是否提供了过路费、海拔变化、注意事项等可执行信息
- 是否考虑了实际驾驶体验（休息点、补给点、信号覆盖）

### E. 整体规划深度

整体方案必须展示**真正的规划思考**，不能只是把景点按地理位置排成清单。

**检查标准：**
- 是否解释了整体路线方向的逻辑（为什么顺时针/逆时针、为什么这个季节走这条路）
- 是否有明确的季节性窗口论证（每个阶段为什么在这个月份最佳）
- 预算分配是否具体到油费/过路费/住宿/餐饮/门票等分项
- 是否考虑了用户特定约束的实际影响（一人行的安全问题、越野车的油耗、10万预算如何分配）
- 是否提供了备选方案（如果某段路封了怎么办、天气不好怎么调整）
- 风险提示是否具体（不是泛泛说"注意安全"，而是具体到哪个路段有什么风险、怎么应对）

### F. 深度偷懒检测

以下行为虽然格式合规，但属于"内容偷懒"：

| 内容偷懒行为 | 检测方法 | 严重程度 |
|---|---|---|
| 地点介绍只有一句定义式描述 | "简单介绍"不足 20 字或只是"XX是XX的XX"句式 | major |
| 推荐理由与用户需求无关 | 推荐理由中未提及用户的任何特定偏好（自然/预算/人群/交通等） | major |
| 时间安排只给范围不给方案 | 多天行程只说"建议X-Y天"但不逐天分配 | major |
| 路线只有起终点无途经细节 | 路线描述只有"A→B，约Xkm"但无关键途经点、服务区、注意事项 | major |
| 预算只给总数无分项 | 预算只说"约X万"但不分油费/住宿/餐饮等 | major |
| 风险提示泛泛而谈 | 只说"注意安全""注意天气"但不具体到路段和应对措施 | major |
| 所有地点推荐理由雷同 | 多个地点使用几乎相同的推荐句式，只是换了地名 | major |
| 无备选方案 | 长途路线无任何备选（天气/封路/身体不适时怎么办） | major |

## 四、输出质量审查（output_quality）

### answer 内容完整性

每个主停留点必须包含以下全部信息（参考 travel-answer-format）：

1. **距离**：距起点/住宿地/上一站多远
2. **推荐交通**：公交/地铁/步行/骑行/打车/自驾，有线路号时写线路号
3. **最多等待**：仅当工具返回时写具体数值，否则如实标注无法确认
4. **路程时间**：总耗时，细分车程/步行/换乘
5. **为什么推荐**：结合用户偏好、顺路逻辑、慢游价值或攻略信号，不能只写"很热门"
6. **简单介绍**：1-2 句介绍地点本身

### 事实与观点标注

answer 必须区分两类信息：

- **高德已确认**：地点名称、地址/区县、坐标、距离、路线、交通耗时、周边 POI
- **攻略/主观信号**：为什么值得去、避坑、人流感受、适合人群、本地体验

未被工具确认的信息必须使用以下标注之一：
- "待实时确认"
- "按当前信息估算"
- "高德未返回该字段"
- "需要补充住宿地后重新优化"

**重要："待实时确认"不是免责符。** 标注"待确认"的同时，必须提供基于攻略素材、常识或经验的可执行建议。例如：
- 错误："漠河→海拉尔约600km（待实时确认）"——只有起终点和距离，用户拿到后无法行动
- 正确："漠河→海拉尔约600km（待实时确认）：建议经满归→根河→额尔古纳，分2-3天。满归有加油站和旅馆，根河是冷极纪念碑所在地可停留1小时。根河→额尔古纳约130km路况较好"——即使距离待确认，途经建议和过夜方案已经可执行

**一票否决：** 如果 answer 中出现"待实时确认"或"按当前信息估算"的标注超过 3 处，且这些标注的路段均未提供途经细节、过夜建议或分段方案，直接判定 content_depth 不合格（cap 在 40 分以下）。

### 禁止的表述

以下表述方式视为伪装未验证事实（违规）：

- "应该""大概""一般""通常" + 具体数据
- "大概有某公交""一般等几分钟""不远""很快到"
- 未标注来源的具体距离、时间、公交线路号、等待时间

### 格式合规

- 必须使用 `### 地点名` 小标题范式
- 不得输出 markdown 代码块、前后缀说明或额外文本（必须只输出单个 JSON object）
- 每个地点介绍控制在 4-6 行左右，避免写成百科
- 不堆表格（除非用户明确要求）
- 每天要有清晰主题（如老街散步、河岸慢行、博物馆半日、美食小环线）

### content_insights 和 route_validation

- `content_insights` 必须包含从攻略素材中提炼的关键体验/避坑/人群适配信号，不能为空（除非未调用知乎）
- `route_validation` 必须列出已验证或仍需确认的地点、距离、路线、交通假设
- route_validation 为空但 answer 中有具体距离/时间数据 = 严重违规（数据来源不明）

## 五、偷懒检测（laziness_detection）

### 偷懒行为清单

| 偷懒行为 | 检测方法 | 严重程度 |
|---|---|---|
| 跳过追问直接给方案 | 首次请求且用户未说"别问"，但 `insufficient_information` 为 false 且无追问记录 | critical |
| 模糊措辞替代具体数据 | answer 中出现"不远""很近""很快""大概X分钟"但无工具验证标注 | major |
| 遗漏关键维度 | 缺少交通、预算、人群适配、拥挤风险、备选方案中的任何一项 | major |
| 内容过短敷衍 | answer 长度明显过短（如 3 天行程 answer 不足 500 字） | major |
| 声称信息不足但未列问题 | `insufficient_information` 为 true 但 `follow_up_questions` 为空 | critical |
| 路线验证为空但有具体数据 | `route_validation` 为空数组但 answer 中有具体距离/时间 | critical |
| 模板化填充 | 所有地点使用完全相同的句式结构，缺乏个性化分析 | minor |
| 照搬热门清单 | 直接堆叠热门景点，无差异化、无本地体验、无替代方案 | major |
| 跳过审查步骤 | 最终输出未经 review-agent 审查 | critical |
| 先写行程再验证 | planning_process 中验证步骤在行程设计之后 | critical |

### 严重程度定义

- **critical**：必须修正，列入 `critical_issues`
- **major**：强烈建议修正，列入 `laziness_flags`
- **minor**：建议改进，列入 `laziness_flags`

## 六、评分规则

### 综合评分计算

```
overall_score = workflow_compliance.score * 0.20
             + thinking_quality.score * 0.15
             + content_depth.score * 0.30
             + output_quality.score * 0.20
             + (100 - laziness_penalty) * 0.15
```

其中 `laziness_penalty` = 每个偷懒行为按严重程度扣分（包括内容偷懒和流程偷懒）：
- critical: -40
- major: -25
- minor: -10

### 一票否决规则（直接判定不合格）

以下任何一条出现，无论其他维度得分多高，overall_score 直接 cap 在 64 分以下：

| 一票否决行为 | 检测方法 |
|---|---|
| "待实时确认"滥用 | 路线/距离/时间标注为"待实时确认"或"按当前信息估算"，但同时未提供任何可执行的途经细节、过夜点、分段建议——用"待确认"当挡箭牌逃避具体内容 |
| 核心路线无实质内容 | 用户请求的核心路线（如"环中国一圈"的主干路线）中，超过 30% 的路段只有起终点和总距离，无途经点、无分段、无过夜建议 |
| 关键决策无论证 | 整体路线方向（顺时针/逆时针）、季节安排、预算分配等关键决策只给结论不给推理过程 |
| 需求追问被跳过 | 用户首次请求未说"别问"但 agent 直接输出完整方案 |

### 各维度评分标准

**workflow_compliance（0-100）：**

| 扣分项 | 扣分 | 具体说明 |
|---|---|---|
| 跳过某一步 | -30/步 | 如跳过追问（-30）、跳过攻略采集（-30）、跳过地理验证（-30） |
| 步骤存在但执行质量差 | -20/步 | 如追问了但没覆盖所有必须缺口；地理验证了但只验证了 1 个点而 answer 引用了 10 个点的数据 |
| 步骤顺序错误 | -25 | 如先写行程再验证、先输出再审查 |
| 追问后继续追问 | -25 | 用户已回复第一轮追问，agent 又输出 insufficient_information: true |
| normalized_query 遗漏用户明确提到的信息 | -15 | 用户说了"越野车"但 normalized_query 没提 |

**thinking_quality（0-100）：**

| 扣分项 | 扣分 | 具体说明 |
|---|---|---|
| 缺少需求识别 | -25 | thinking_result 未提及用户的目的地/天数/交通偏好等关键字段 |
| 缺少信息缺口判断 | -25 | 未分析"我还缺什么信息"就直接进入采集/规划 |
| 缺少工具调用依据 | -25 | 调用了 zhihu_guide_material 或 amap-agent 但未说明为什么调用 |
| 缺少路线取舍 | -25 | 只列了选择结果，未说明"为什么选 A 不选 B" |
| 偷懒推理 | -30 | 跳过缺口分析直接假设默认值；声称"已验证"但 route_validation 无对应记录 |
| 暴露隐藏推理链路 | -15 | thinking_result 中出现内部实现细节或逐字推理 |
| thinking_result 与 planning_process 高度重复 | -15 | 两者内容几乎相同，无推理/执行的区分 |

**content_depth（0-100）：**

| 扣分项 | 扣分 | 具体说明与示例 |
|---|---|---|
| 地点介绍只有一句定义式描述 | -15/个 | "漠河北极村是中国最北点"→ 缺少去了看什么、什么时间段去、和北红村怎么选 |
| 推荐理由与用户需求无关 | -15/个 | "自然风光壮美，值得一去"→ 未关联用户"自然为主+一人越野车+10万预算"的特定需求 |
| 时间安排只给范围不给方案 | -20/阶段 | "建议分3-5天"→ 用户无法据此订酒店、安排补给、规划每日驾驶时长 |
| 路线只有起终点无途经细节 | -15/个 | "漠河→海拉尔约600km"→ 未给出途经满归/根河的具体建议、未标注加油站间隔、未说明在哪过夜 |
| 预算只给总数无分项 | -15 | "约10万"→ 用户无法判断油费是否算对、住宿标准是否合理 |
| 无备选方案 | -15 | 长途路线无 Plan B（独库封路怎么绕、丙察察走不了怎么办） |
| 风险提示泛泛而谈 | -10 | "注意高反"→ 未说明哪个路段海拔骤升、症状判断标准、应急下撤路线 |
| 推荐理由雷同 | -25 | 多个地点都用"XX是XX的代表，值得一去"句式，只是换了地名 |
| 未解释整体路线方向逻辑 | -15 | 长途路线未说明为什么顺时针而非逆时针 |
| 季节性窗口无论证 | -15 | 涉及多季节行程但未论证"为什么6月走东北、9月走新疆" |
| 无逐阶段预算分配 | -10 | 有总预算分项但未按阶段拆分，用户无法判断哪个阶段最烧钱 |
| 无城市休整点标注 | -10 | 长途路线未标注哪些节点适合进城补给、休整、维修车辆 |

**output_quality（0-100）：**

| 扣分项 | 扣分 | 具体说明 |
|---|---|---|
| 一个停留点缺六要素之一 | -15 | 缺距离/推荐交通/最多等待/路程时间/为什么推荐/简单介绍 |
| 一处事实/观点未标注 | -25 | answer 中出现具体数据但未区分"高德已确认"和"攻略/主观信号" |
| 一处伪装未验证事实 | -30 | 用"应该""大概""一般""通常" + 具体数据来规避标注 |
| 格式严重不符 | -20 | 未使用 `### 地点名` 小标题、输出了 markdown 代码块等 |
| "待实时确认"无补充内容 | -30/处 | 标注"待实时确认"但未同时给出途经点、过夜建议、分段方案等可执行信息——"待确认"不是免责符 |

**laziness_detection（0-100）：**
- 从 100 开始，按偷懒行为扣分

### 评分区间含义

- 90-100：优秀，可直接输出
- 85-89：良好，建议改进
- 75-84：合格，修正后输出
- 65-74：不合格，必须修正后重新审查
- 0-64：严重不合格，必须大幅修改
