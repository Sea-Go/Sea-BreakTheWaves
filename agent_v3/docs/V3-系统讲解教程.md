# V3 系统讲解教程

这篇文档按真实代码链路来拆 V3，不讲大而化之的“Agent 架构理念”，而是直接回答三个问题：

- 一个请求是怎么在系统里跑起来的。
- skill、tool、workflow、graph、review 是怎么分工的。
- 新人如果要扩一个能力，应该从哪里下手。

阅读建议：

- 先通读本文，建立全链路脑图。
- 再对照 [main.go](../main.go)、[TravelPlanningAgent.go](../agent/TravelPlanningAgent.go)、[workflow_runner.go](../agent/workflow_runner.go) 看代码。
- 如果你只想先抓主线，优先看第一章、第二章和第五章。

## 第一章：从入口看全链路

### 先认识一下 `trpc-agent-go`

在这个项目里，`trpc-agent-go` 不是一个“顺手接进来的模型 SDK”，而是整套 Agent 系统的运行时框架。

可以先把它理解成下面这层分工：

- 它负责提供 Agent 基础设施：`llmagent`、`team`、`runner`、`skill`、`tool`、`agui`、`session`、`memory`、`telemetry`。
- 它不替你定义业务流程：比如旅行规划要分几阶段、哪些节点要进图数据库、哪些结果要审查，这些仍然是项目代码自己决定。

所以学习这个项目时，最好先建立一个判断：

- 看到 `trpc-agent-go` 相关代码时，优先问“这是框架提供的通用能力”。
- 看到 `agent/`、`graph/`、`tools/`、`skills/` 里的项目代码时，再问“这是业务层在底座上加的规则和实现”。

有了这层区分，再往下看代码会轻松很多，因为你不会把“框架职责”和“项目职责”混在一起。

先只盯三份文件：

- [main.go](../main.go)
- [TravelPlanningAgent.go](../agent/TravelPlanningAgent.go)
- [workflow_runner.go](../agent/workflow_runner.go)

主链路可以先压缩成下面这条：

```text
HTTP / AGUI 请求
  -> runner
  -> TravelPlanningAgent
  -> TravelSkillOrchestrator
  -> graphWorkflowAgent
  -> stage-by-stage execution
  -> final JSON / streaming output
```

### 1. `main.go` 负责启动系统

在 [main.go](../main.go) 里，能看到几个关键动作：

- 初始化日志。
- 初始化 metric 和 trace。
- 初始化配置和 Neo4j 连接。
- 通过 `agent.NewTravelPlanningAGUIHandler()` 创建 AG-UI / SSE 入口。
- 暴露 `/agui` 和 `/travel/` 路由。

这里的重点不是“开了个 HTTP 服务”，而是它把 `trpc-agent-go` 的运行时、流式输出和观测体系都接好了。对新人来说，这能建立一个重要认知：Agent 系统不是只有模型调用，它首先是一个服务。

### 2. `TravelPlanningAgent.go` 负责选择工作模式

[TravelPlanningAgent.go](../agent/TravelPlanningAgent.go) 里最关键的判断是：

- 如果 Neo4j 可用，就走 `NewGraphWorkflowAgent()`。
- 如果 Neo4j 不可用，就退回传统 Team coordinator。

这体现了项目里的一个核心设计：图数据库不是可有可无的装饰，而是 V3 正式工作流的主状态源。

同一个文件里还能看到 `trpc-agent-go` 的几层落地：

- `llmagent` 定义 Agent。
- `team` 组织多 Agent 协作。
- `runner` 负责运行一次请求。
- `session` 和 `memory` 提供短期会话与长期记忆。
- `agui` 把运行结果流式返回给前端。

### 3. `workflow_runner.go` 负责把阶段真正跑完

[workflow_runner.go](../agent/workflow_runner.go) 是整个系统最值得新人反复阅读的文件，因为它体现了“工程化 Agent”和“单个 prompt”的根本区别。

它做的事不是“给模型一个大 prompt，然后等结果”，而是：

- 先从 orchestrator 拿到当前阶段。
- 根据阶段选择对应处理逻辑。
- 每一阶段完成后更新 runtime。
- 在合适的 checkpoint 停下来或者继续推进。
- 在错误发生时输出结构化错误和 trace。

这就是典型的“LLM 负责生成与判断，Go 负责流程控制与真相校验”的架构。

## 第二章：工作流如何体现

V3 当前主链路围绕 `TravelSkillStage` 展开，定义见 [travel_skill_types.go](../agent/travel_skill_types.go)。

### 阶段总览

| 阶段 | 作用 | 谁负责 |
|------|------|--------|
| `requirement_intake` | 抽取需求字段，识别缺失信息 | `TravelSkillOrchestrator` + `IntakeAgent` |
| `awaiting_user_info` | 停住等待用户补充 | `TravelSkillOrchestrator` |
| `requirement_merge` | 合并用户补充内容 | `TravelSkillOrchestrator` + `IntakeAgent` |
| `macro_planning` | 创建 `TripPlan + Phase` | `graphWorkflowAgent` + `MacroPlanningAgent` |
| `graph_splitting` | 把阶段拆到 `Month/Week/Day` | Go 代码 |
| `day_expansion` | 逐日补 POI、路线、天气验证 | Go 代码 + 地图工具 / Agent |
| `review` | 多层审查 | Review Agents + 图写回 |
| `final_output` | 生成最终输出 | Go 代码 + 输出 Agent |

### `TravelSkillOrchestrator` 为什么重要

[travel_skill_orchestrator.go](../agent/travel_skill_orchestrator.go) 是整个系统的阶段控制中心。它决定：

- 当前 session 处于哪个阶段。
- 什么时候该追问用户。
- 什么时候可以进入正式规划。
- 什么时候要重置 runtime，开始一轮新的规划。

这点非常关键：阶段推进权不在 LLM 手里，而在 Go 层 orchestrator 手里。

如果把这件事完全交给 prompt，系统往往会出现三类问题：

- 该停的时候不停，继续乱生成。
- 该追问的时候不追问，直接瞎补默认值。
- 同一个用户多轮对话后，当前阶段漂移，最后谁也不知道系统现在在干什么。

### 三个必须理解的核心类型

这三个类型是新人阅读这个项目时必须先建立的脑图。

#### 1. `TravelSkillStage`

它定义了系统当前处在哪个阶段。你可以把它理解成“状态机枚举值”。

它的意义不在于常量本身，而在于：所有流程决策都围绕它展开。只要 `CurrentStage` 没变，系统的工作内容就不该变。

#### 2. `TravelRequirementSnapshot`

它是用户需求的结构化快照，包含：

- 行程范围和天数。
- 出发地和日期。
- 预算、交通方式、节奏、偏好。
- `must_visit`、`avoid_places`、`destination_anchors`。

这个类型的重要性在于：需求不是一次性抽出来就结束，而是在 `intake` 和 `merge` 阶段逐步填充的。

#### 3. `SkillResult`

它是 skill 执行后的标准输出协议，告诉 orchestrator：

- 现在是否 ready。
- 缺了哪些字段。
- 要不要停下来等用户。
- 下一步该进入哪个 stage。

这个结构把“模型输出”变成了“程序可以消费的控制信号”，这是工程化 Agent 非常重要的一步。

## 第三章：图数据库怎么做

### 为什么这里要用 Neo4j

如果只是做一个 1 到 3 天的小行程，很多信息还可以暂时留在上下文里。但这个项目面向的是多阶段、长链路规划，光靠上下文会很快出现问题：

- 数据越来越长，模型窗口越来越贵。
- 局部修改困难，很难只改其中一个阶段或某一天。
- 很难做审查挂载、状态追踪和增量补写。

所以 V3 把 Neo4j 当成“计划状态源”，而不是只当存档库。

### 图模型长什么样

当前核心层级是：

```text
TripPlan
  -> Phase
  -> Month
  -> Week
  -> Day
  -> POI / Route / Review / GuideInsight / ClimateData
```

在 [graph/models.go](../graph/models.go) 里可以看到主要节点类型：

- `TripPlanNode`
- `PhaseNode`
- `MonthNode`
- `WeekNode`
- `DayNode`
- `POINode`
- `RouteSegmentNode`
- `GuideInsightNode`
- `ReviewResultNode`
- `ClimateDataNode`

这个设计的核心价值是：它让系统可以按层级增量演进，而不是每次全量重写。

### 图读写是怎么分层的

图相关能力在 `tools/graph_*` 和 `graph/` 目录里分成两层：

- `graph/`：真正的 Neo4j 读写实现。
- `tools/`：把这些能力包装成 Agent 可调用的 tool。

比如：

- [graph_write_tools.go](../tools/graph_write_tools.go) 提供写工具。
- [graph_read_tools.go](../tools/graph_read_tools.go) 提供读工具。
- [graph_tools.go](../tools/graph_tools.go) 负责把工具组装成 `GraphToolSet`。

### 三个最值得新人先看的关键工具

#### `create_trip_plan`

作用：创建根节点，并绑定 `userId / sessionId / requestId`。

它的重要点不在“创建一个节点”，而在“TripPlanID 所有权回到 Go 层”。系统先预生成 ID，再交给 Agent 使用，最后回到数据库核验，而不是信任 LLM 在文本里说它创建成功了。

#### `split_parent_node`

作用：把父节点拆成子节点，比如：

- `TripPlan -> Phase`
- `Phase -> Month`
- `Month -> Week`
- `Week -> Day`

它的价值在于：把“层级拆分”抽成一个统一接口，而不是每一层都单独写一套完全不同的工具协议。

#### `get_day_full_context`

作用：只读取某一天完整子图，包括 `Day + POIs + Routes + Insights + Weather + Reviews`。

这体现了一个非常重要的工程思想：最小上下文加载。处理哪一天，就只读取哪一天，而不是把全年计划全塞回来。

### 这里的检索增强怎么理解

本项目更适合表述成“图上下文检索增强”或“graph-backed retrieval”，而不是标准向量 RAG。

当前代码里的检索增强包括：

- 通过攻略工具采集素材。
- 通过 `write_guide_insight` 把 `GuideInsight` 写入图。
- 通过 `get_subgraph`、`get_day_full_context`、`get_trip_overview` 做局部上下文读取。
- 基于图上下文生成 day/week/final output。

它的重点是“先把事实和证据写入状态图，再按需加载回来”，不是“先嵌入，再相似度召回”。

## 第四章：Agent、检索增强和模式

### 1. Agent 分工模式

这个项目不是一个万能 Agent 包打天下，而是典型的“专职 Agent + 编排器”模式。

主要分工如下：

- `IntakeAgent`：做需求准入和字段抽取。
- `MacroPlanningAgent`：只做 `TripPlan + Phase` 级别宏观规划。
- `AmapAgent`：负责地理事实、POI 和路线验证。
- `Dili360Agent`：负责区域背景和地理百科信息。
- `Review Agents`：负责从 workflow、thinking、content、output 等角度审查结果。

这套分工对新人最重要的启发是：不要让一个 Agent 同时负责“理解需求、查事实、规划路线、审查结果、生成最终文案”，那样系统很快会失控。

### 2. 检索增强模式

这个项目里的检索增强不是一个独立模块，而是贯穿在多个阶段里的：

- 宏观阶段会引入区域气候和地理背景。
- 中间阶段会采集攻略素材并写入 `GuideInsight`。
- 微观阶段会按 `Day/Week/Phase` 读取局部子图。
- 输出阶段会基于图里的已验证上下文生成文案。

所以这里更好的说法是：先做外部信息采集，再做图内证据沉淀，最后按需局部检索回来。

### 3. 这个项目里能学到的几个模式

- Orchestrator 模式：流程推进由 Go 控制。
- Specialized Agent 模式：不同 Agent 专注不同能力。
- Graph State 模式：把计划状态放进图，而不是全部留在 prompt。
- Review Gate 模式：生成与审查分离。
- Streaming Output 模式：结果以 AG-UI / SSE 流式返回。

这些模式才是这个项目真正适合做课程的地方。

## 第五章：skill 和 tool 怎么创建

这一章最适合新人动手实操。

### skill 怎么写

skill 存在于 `skills/` 目录里，本质上是“规则和协议文档”。

以 [travel-requirement-intake/SKILL.md](../skills/travel-requirement-intake/SKILL.md) 为例，一个 skill 通常包含：

- frontmatter：名字和用途描述。
- 职责说明：它该做什么。
- 禁止事项：它不能做什么。
- 输入输出协议：必须输出什么 JSON。
- 示例：给模型一个稳定参照。

最小骨架通常像这样：

```md
---
name: example-skill
description: 这个 skill 解决什么问题
---

# 目标

说明职责和边界。

# 禁止

- 不要调用某类工具
- 不要直接生成最终答案

# 输出协议

只输出单个 JSON object。
```

对新人最重要的理解是：skill 主要负责“规则、边界、协议、模板”，不是负责执行代码。

### tool 怎么写

tool 存在于 `tools/` 目录里，本质上是“可执行能力”。

以 [graph_write_tools.go](../tools/graph_write_tools.go) 为例，一个 tool 的写法通常分三步：

1. 定义 typed input/output struct。
2. 给字段补 `jsonschema` 描述。
3. 用 `function.NewFunctionTool` 包装执行函数。

最小结构大致如下：

```go
type ExampleInput struct {
    Name string `json:"name" jsonschema:"required,description=名称"`
}

type ExampleOutput struct {
    Success bool `json:"success"`
}

func newExampleTool(...) tool.Tool {
    return function.NewFunctionTool(
        func(ctx context.Context, in ExampleInput) (ExampleOutput, error) {
            return ExampleOutput{Success: true}, nil
        },
        function.WithName("example_tool"),
        function.WithDescription("示例工具"),
    )
}
```

### 为什么要强调 typed struct 和 `jsonschema`

因为这会直接影响：

- 模型能不能更稳定地理解参数。
- 调用失败时能不能快速定位原因。
- 文档和代码能不能保持一致。

### skill 和 tool 的契约边界

这是新人最容易混淆的地方。

- skill 偏规则与编排协议。
- tool 偏执行能力与结构化 I/O。

可以这样记：

- “应该怎么做、不能怎么做、输出长什么样”写进 skill。
- “真的去查、真的去写、真的去调外部系统”写进 tool。

如果一个文件同时承担两种职责，系统通常很快会变得难维护。

## 第六章：我们怎么做规范

这套项目最值得保留下来的，其实是一套工程规范。

### 1. 阶段推进只在 Go 层

阶段控制权只在 orchestrator，不交给 LLM 自由决定。

### 2. TripPlanID 由 Go 预生成

ID 先由程序生成，再由 Agent 使用，再由数据库核验，避免把核心事实建立在 LLM 自述上。

### 3. Database truth first

最终真相以数据库为准，而不是以模型文本为准。

### 4. 最小上下文加载

处理某个节点时，只加载这个节点的局部子图，不把全量数据都读回来。

### 5. 审查和生成分离

先生成，再由专门的 review agent 或 review tool 检查，不把“自我审查”完全交给同一个 Agent。

### 6. 工具返回结构化 JSON

tool 的输入输出必须尽量稳定、可判定、可追踪。

### 7. skill 不越权

skill 可以约束行为、规定协议，但不应该偷偷承担 tool 的执行职责。

## 第七章：新人扩展示例

这里给两个适合作为练习题的扩展方向。

### 练习 1：新增一个 skill

目标：新增 `travel-budget-review` skill，用来检查月度预算和阶段预算是否失衡。

建议步骤：

1. 在 `skills/` 下新建目录和 `SKILL.md`。
2. 写清楚职责、禁止事项、评分标准和 JSON 输出协议。
3. 让某个 review agent 通过 `skill_load` 使用它。

练习重点：

- 你是否真正理解了 skill 的职责边界。
- 你是否会用结构化输出协议跟 orchestrator / review 流程对齐。

### 练习 2：新增一个 tool

目标：新增 `get_trip_phase_summary` tool，返回某个 `TripPlan` 下所有 `Phase` 的摘要信息。

建议步骤：

1. 在 `tools/` 里定义 input/output struct。
2. 在 `graph/` 里补对应查询逻辑。
3. 用 `function.NewFunctionTool` 暴露为可调用工具。
4. 挂到合适的 tool set 里。

练习重点：

- 你是否理解 tool 的 typed I/O 设计。
- 你是否知道 graph 层和 tool 层应该如何分层。

## 结语

如果你是新人，这个项目最值得学的不是“它会规划旅行”，而是它把 Agent 工程里最容易失控的几个问题都拆开了：

- 谁控制阶段。
- 谁负责事实。
- 状态放在哪里。
- 证据怎么沉淀。
- 输出怎么审查。

你能把这几个问题讲明白，再去做别的 Agent 项目，会轻松很多。
